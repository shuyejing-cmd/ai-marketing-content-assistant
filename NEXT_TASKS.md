## P0 账号注册系统与多用户隔离
目标：实现邮箱密码注册/登录，让多个用户使用所有功能时数据互不干扰。这是后续素材库、模板管理、角色权限、支付和团队能力的基础。

第一版默认方案：

- 邮箱 + 密码注册。
- 邮箱 + 密码登录。
- 支持退出登录。
- 使用服务端 session/cookie 保持登录状态。
- 保留匿名 `ownerId`，但登录后由 `User` 接管数据归属。

验收标准：

- 用户可以注册、登录、退出。
- 密码只保存安全 hash，不保存明文。
- 登录用户只能看到自己的 Session、GenerationTask、GenerationResult、ImageAsset、PromptLog 关联数据。
- 自由会话和模板会话在用户维度继续隔离。
- 新用户看不到旧匿名用户或其他用户的历史会话。
- 匿名 `ownerId` 有明确迁移策略：用户登录/注册时可把当前浏览器匿名数据绑定到该账号，或明确选择不迁移。
- 模板管理权限从 `TEMPLATE_ADMIN_SECRET` 过渡到账号角色，第一版至少支持 `admin` / `user`。
- 管理员可以继续访问 `/admin/templates`，普通用户不能访问模板内部 prompt。
- API 层不再信任前端传来的 owner 身份；服务端从登录态解析当前用户。
- E2E 覆盖至少两个用户互不串数据。

建议拆分任务：

1. 账号数据模型和迁移方案。
2. 密码 hash、注册、登录、退出 API。
3. 当前用户解析和 API owner 来源替换。
4. 匿名数据迁移到账号。
5. 模板管理角色权限。
6. 前端登录/注册/退出 UI。
7. 多用户隔离单测和 E2E。

## P0 真实模型链路持续实机复验
目标：保持 APIMart 图片、COS 图生图中转和 Ark 文案模型在真实环境中可用。

验收标准：

- 文生图能返回 APIMart 远程模型图。
- 上传图生图能上传 COS，并把私有签名 URL 传给 APIMart。
- Ark 文案模型返回真实 `title`、`publishingCopy`、`imageText`，日志显示 `generation.copy_provider.success`。
- 下载按钮下载的是模型原图，不是 canvas fallback 模板。
- 服务端日志不输出真实密钥、不输出完整 COS 签名 URL。

## P1 模板版本管理
目标：让模板 prompt 修改可追踪、可回滚、可比较。

验收标准：

- 模板内部 prompt 有版本记录或明确版本字段方案。
- PromptLog 能区分自由生成、模板生成、重新生成、二次修改。
- 能从数据库或日志复盘一次生成的完整输入。

## P1 模板效果评估记录
目标：为后续模板调优提供依据。

验收标准：

- 记录模板 id、任务 id、是否下载、是否复制文案、是否重新生成。
- 不先做复杂数据看板。
- 不影响当前生成主流程。

## P1 图片资产长期对象存储迁移
目标：避免 PostgreSQL 长期存储大量 base64 图片。

验收标准：

- 设计 `ImageAsset` 从 base64 迁移到对象存储 URL/key 的方案。
- 保留本地开发 fallback。
- 不破坏已有生成结果读取。
- 明确对象存储未配置时的行为。

## P1 第一批真实图片模板运营
目标：创建并调优一批可实际使用的图片模板。

验收标准：

- 至少创建 3 个发布状态图片模板。
- 每个模板包含封面、标题、描述、内部 prompt。
- 同一商品图反复生成时，模板风格和构图方向稳定。
- 普通用户不能看到内部 prompt。

## P2 NestJS 后端迁移评估
目标：判断何时从 `apps/web` API Routes 迁移到 NestJS 主后端。

验收标准：

- 账号系统和当前图片 MVP 稳定后再评估。
- 明确迁移模块顺序：用户、会话、任务、模板、素材、PromptLog。
- 明确迁移期间 API 合同如何保持兼容。
- FastAPI 仍只作为后期 AI 执行服务方向，不承载主业务状态。
